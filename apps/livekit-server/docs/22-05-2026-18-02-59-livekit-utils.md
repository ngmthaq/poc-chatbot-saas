## Approach Summary

Install `livekit-server-sdk` as a dependency of the `livekit-server` workspace, extend `config/env.ts` to expose a typed `livekit` block (apiKey/apiSecret/url) read once at startup with fail-fast validation, and add a `.env.example` row per key (names only). Build three small singleton-exported util classes — `LiveKitTokenUtil`, `LiveKitRoomUtil`, `LiveKitParticipantUtil` — mirroring the existing `HealthService` pattern. Each util reads from `config.livekit` (never `process.env` directly), wraps the SDK's `AccessToken` / `RoomServiceClient`, and exposes a focused set of methods (SRP per file). The `RoomServiceClient` is instantiated once per util in the constructor (DRY/KISS).

Verified via LiveKit Docs MCP:

- Package: `livekit-server-sdk` (Node.js)
- `AccessToken`, `at.addGrant({ roomJoin, room, ... })`, `at.toJwt()`
- `RoomServiceClient` → `createRoom`, `listRooms`, `deleteRoom`, `updateRoomMetadata`
- `listParticipants`, `getParticipant`, `removeParticipant`, `updateParticipant(room, identity, metadata?, permission?)`, `mutePublishedTrack`
- `getRoom` is NOT a first-class API → implemented via `listRooms({ names: [name] })`

## Functional Requirements

1. `livekit-server-sdk` installed as runtime dep of `apps/livekit-server`.
2. `.env.example` lists `LIVEKIT_API_KEY=`, `LIVEKIT_API_SECRET=`, `LIVEKIT_URL=`.
3. `config.livekit.{apiKey, apiSecret, url}` exposed by `src/config/env.ts` with fail-fast (key-NAME-only error).
4. `livekit-token.ts` — `LiveKitTokenUtil` + `livekitTokenUtil` singleton with `createAccessToken(...)`.
5. `livekit-room.ts` — `LiveKitRoomUtil` + `livekitRoomUtil` with `createRoom`, `listRooms`, `getRoom`, `deleteRoom`, `updateRoomMetadata`.
6. `livekit-participant.ts` — `LiveKitParticipantUtil` + `livekitParticipantUtil` with `listParticipants`, `getParticipant`, `removeParticipant`, `updateParticipantMetadata`, `mutePublishedTrack`, `updateParticipantPermissions`.
7. Each util reads from `config.livekit` only (not `process.env`).
8. No routes/controllers/services/middlewares introduced.

## Non-Functional Requirements

- TS strict-compatible; `import type` for type-only imports (verbatimModuleSyntax).
- Class + singleton export pattern matches `HealthService`/`HealthController`.
- No logging or string-interpolating secret VALUES anywhere; validation errors reference env-key NAMES only.
- No network calls on module import; clients built once in constructor.
- Thin wrappers — no retries/caches/extra abstractions.
