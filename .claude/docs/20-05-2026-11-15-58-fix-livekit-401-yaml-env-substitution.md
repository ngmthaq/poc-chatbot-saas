## Approach Summary

- The credential-drift hypothesis is falsified by the operator (hashes match across all three `.env` files). Reading the upstream `livekit/livekit` source (`cmd/server/main.go::getConfigString` and `pkg/config/config.go::NewConfig`) shows the YAML body is loaded via `os.ReadFile` and decoded with `yaml.NewDecoder` directly — there is no `os.ExpandEnv` over the YAML, and `os.ExpandEnv` is only ever applied to `conf.KeyFile`. The `LiveKit server image v1.7.2` therefore stores the **literal** strings `${LIVEKIT_API_KEY}` and `${LIVEKIT_API_SECRET}` in its `Keys` map. Every worker JWT signed with the real secret then fails HMAC validation, producing the 401 on the agent-worker's `/agent` WebSocket Upgrade.
- The misleading comment block at the top of `livekit-server/livekit.yaml` ("Environment variables (`${VAR}`) are substituted by LiveKit itself at startup") was the trap that led to the original wrong credential-drift theory. The comment is factually wrong against upstream behavior; only the official `LIVEKIT_CONFIG` (whole-body) and `LIVEKIT_KEYS` ("key: secret" string) env vars are honored.
- The minimum-blast-radius fix that satisfies the operator's "align only / no rotation" constraint is to feed the keys via `LIVEKIT_KEYS` (already supported by `livekit/livekit-server:v1.7.2` per `cmd/server/main.go` `--keys` flag with `Sources: cli.EnvVars("LIVEKIT_KEYS")`). `LIVEKIT_KEYS` accepts the exact string `"<key>: <secret>"`, parsed by `unmarshalKeys` via `yaml.Unmarshal`. Source the value from the already-present `LIVEKIT_API_KEY` / `LIVEKIT_API_SECRET` in `livekit-server/.env` using shell-form compose `command:` (analogous to the existing redis `sh -c "exec redis-server ... --requirepass \"$$REDIS_PASSWORD\""` pattern in the same compose file), with no new secret generation.
- Concurrently, `livekit.yaml` is updated to drop the broken `keys:` block (now provided via `--keys`/`LIVEKIT_KEYS`) and to inline the literal `redis.password` via the same env-var-at-shell-startup pattern (or move Redis to `redis-host`/`redis-password` CLI flags which `cmd/server/main.go` exposes with `Sources: cli.EnvVars("REDIS_PASSWORD")` — note: server already reads `REDIS_PASSWORD` natively, so the `redis.password:` line in YAML is also broken today and the CLI flag is the cleanest replacement). The misleading top-of-file comment must be replaced with a correct one explaining the actual substitution semantics.
- The operator's downstream preferences (`LIVEKIT_URL=ws://host.docker.internal:7880`, three `.env` files synced by a script) are kept compatible but are not the primary diagnostic concern: the worker DID complete the TCP handshake to the server (otherwise the error would be `ECONNREFUSED`, not `Unexpected server response: 401`), proving network reachability and IP routing are not the root cause. Fixing the URL alone would NOT have fixed the 401.

## Risks & Assumptions

- Assumption: `livekit/livekit-server:v1.7.2` honors `LIVEKIT_KEYS` and `--keys` per the master-branch source we inspected. Likely true (the flag has been stable for years), but the developer should verify against the v1.7.2 tag specifically (`mcp__livekit-docs__get_changelog` for `livekit/livekit` if needed). If `LIVEKIT_KEYS` was renamed in v1.7.x, fall back to setting `LIVEKIT_CONFIG` (whole-YAML-body env var) — also supported by the same source file.
- Assumption: docker-compose's own `${VAR}` substitution (which IS performed by `docker compose` at parse time, NOT by LiveKit) will pick up values from `livekit-server/.env` automatically because compose v2 loads the project-directory `.env` for compose-level interpolation. This means option B (`environment: LIVEKIT_KEYS: "${LIVEKIT_API_KEY}: ${LIVEKIT_API_SECRET}"`) is simpler than option A (shell-form `command:`). Both work; recommend option B for clarity.
- Risk (backward compatibility): backend-service mints user/room JWTs with the same API key/secret, and validates webhooks signed by the LiveKit server (`livekit.config.ts` → `webhookPath`). Once the server's `Keys` map actually contains real credentials, **both flows transition from "broken" to "working" simultaneously** — there is no regression because token-minting against a broken server today already returns rejected JWTs from the server side. Webhook signatures use the same shared secret, so they too start verifying correctly post-fix.
- Risk (musl→glibc and zod fixes regression): both prior cycles' fixes are in `agent-worker/Dockerfile` and `agent-worker/package.json` / `yarn.lock`. This plan does NOT modify any of those files (with the exception of `.env.example` and `README.md`, which were ALREADY touched safely in cycle 1) — preserves both fixes.
- Risk (Redis password): the existing `redis.password: ${REDIS_PASSWORD}` line in `livekit.yaml` ALSO stores the literal placeholder string today. Redis itself has been refusing the LiveKit server's auth attempts since project start — but Redis is only used for multi-node messaging, and the LiveKit server falls back to local-only routing on a single-node deploy. Removing this line and relying on the server's native `REDIS_PASSWORD` env var fixes a latent bug that nobody filed. **Call out to operator that this is in-scope.**
- Risk (silent failure mode): the LiveKit server with placeholder keys logs `"secret is too short, should be at least 32 characters for security"` at startup but does NOT exit (see `ValidateKeys` in `pkg/config/config.go` — only `logger.Errorw`, not `return err`). - Risk (operator's three-file sync script): the operator asked for "three `.env` files synced by a script". Including the script in this fix is out of scope of the 401 root cause and risks scope creep; recommend `SKIPPED` in this iteration and ticketed separately as a developer-experience task.

Add to verification checklist.
- Risk (LiveKit server NOT restarted after fix): updating `livekit-server/.env` or `docker-compose.yml` requires `docker compose up -d --force-recreate livekit` — `docker compose restart` is NOT enough to re-read env. Explicit step in verification.
- Risk (assumption that the binary path in the v1.7.2 image is what option A's shell-form `command:` would need): if developer picks option A, verify path with `docker run --rm --entrypoint sh livekit/livekit-server:v1.7.2 -c "which livekit-server"` before committing. Option B avoids this entirely.
