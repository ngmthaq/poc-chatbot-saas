# Resolve Pull Request

Turns a PR's **reviewer comments** into resolved code changes. It fetches the inline review comments via **MCP**, falling back to the platform **CLI** (`gh`, `glab`, or a Bitbucket CLI), triages them into actionable tasks, then delegates the fixes through the **party-mode** orchestration workflow (Root Agent → developer/tester sub-agents → review → report). Once changes are accepted, it replies to and resolves each addressed comment thread.

## How to Use This Skill

1. Resolve the access method (MCP or CLI), then the target PR.
2. Fetch every reviewer comment with its file, line, body, author, and thread/resolved state.
3. Triage and group the comments into actionable tasks (skipping non-actionable or already-resolved ones).
4. Hand the grouped tasks to **party-mode** — the Root Agent brainstorms, plans, gets approval, delegates to developer/tester sub-agents, and reviews.
5. After fixes are accepted, push the branch, then reply to and resolve each addressed comment thread.
6. Report what was resolved, what was skipped (with reasons), and what remains.

## Notes

- This skill **modifies code** — but never directly. All implementation is delegated to party-mode's developer/tester sub-agents; the Root Agent never edits files itself.
- The party-mode **approval gate is non-negotiable**: no fix is applied until the user approves the resolution plan.
- Only act on comments you can tie to a concrete file + line and a clear, actionable request. Skip resolved threads, outdated comments, pure praise, and vague notes — and report why each was skipped.
- Never expose secrets or credentials surfaced in a comment thread; redact them as `[REDACTED]`.
- Never add AI/agent attribution to thread replies (no "Generated by", "Resolved by Claude", "Co-Authored-By: Claude", or similar). Each reply states only what changed.
