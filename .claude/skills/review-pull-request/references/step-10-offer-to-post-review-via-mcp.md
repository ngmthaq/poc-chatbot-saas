# Step 10 — Offer to Post Review (File + Line Comments)

After displaying the report, ask:

> _"Should I post these findings as file + line comments on PR #[number]?"_

Each finding is posted as an **inline comment anchored to a specific file and line**. Do **not** post a general PR-level / summary comment — every comment must target a file + line.

```markdown
Review Comments:

| File           | Line | Comment                                                                                        |
| -------------- | ---- | ---------------------------------------------------------------------------------------------- |
| `file.ts`      | 42   | "SRP violation: this function does too much. Need to refactor into smaller functions."         |
| `file.ts`      | 88   | "Avoid interpolating user input directly into SQL queries. Use parameterized queries instead." |
| `file.spec.ts` | 40   | "AAA violation: test name describes implementation details rather than behavior."              |
```

Skip any finding that cannot be tied to a concrete file + line rather than promoting it to a general comment.

Do **not** add any AI/agent attribution to the comment body — no "Generated by", "Reviewed by Claude", "Co-Authored-By: Claude", or similar tool/author signature lines. Each comment contains only the finding itself.

## Posting via MCP

If confirmed, use the MCP tool to submit each finding as a **line-level review comment** on the corresponding file and line. Prefer submitting as a single **review** with inline annotations over individual comments where the MCP supports it. Never post a top-level PR comment.

## Posting via CLI

If only a CLI is available, post one inline comment per finding via the platform's API-backed command. These commands require the file path, line, and (for GitHub) the head commit SHA.

**GitHub** — use the REST API through `gh` for line-anchored comments (the plain `gh pr comment` posts a PR-level comment and must **not** be used here):

```bash
gh api repos/{owner}/{repo}/pulls/{number}/comments \
  -f body="<comment>" \
  -f commit_id="<head-sha>" \
  -f path="<file>" \
  -F line=<line> \
  -f side="RIGHT"
```

Get the head SHA with `gh pr view <number> --json headRefOid -q .headRefOid`.

**GitLab** — post a discussion anchored to a diff line:

```bash
glab api projects/:id/merge_requests/<number>/discussions \
  -f body="<comment>" \
  -f "position[position_type]=text" \
  -f "position[new_path]=<file>" \
  -f "position[new_line]=<line>" \
  -f "position[base_sha]=<base-sha>" \
  -f "position[head_sha]=<head-sha>" \
  -f "position[start_sha]=<base-sha>"
```

**Bitbucket** — use the installed CLI's inline-comment command (flags vary by tool); it must accept a file path and line number. If the CLI cannot anchor a comment to a file + line, do not fall back to a general comment — report the findings to the user as the table above instead.

After posting, report back how many inline comments were created and the PR/MR URL.
